# SimsCommu: 一个以语音开始、以语音结束的口语练习实验

SimsCommu 是一个多 Agent 口语练习 Web 应用。这个项目的出发点其实很简单：一是我个人想练习口语，二是我想验证 STT、TTT、TTS 这条三段式语音架构，能不能被组织成一个真正可用的 AI 应用。

## 项目状态：已中止 · 等待重启

我决定中止当前版本。原因很直接：相比 STT -> TTT -> TTS 的串行拼接，Live 双工模型是更好的实现方式。它从底层就面向实时语音交互，能自然支持双方随时插话、打断和继续，而这些恰恰是口语练习体验的核心。

现有实现和实验结论会被保留，但不再沿着当前架构继续投入。项目进入等待重启状态；未来若重启，将以 Live 双工模型为新的技术起点，而不是继续扩建现有的三段式链路。

## Background

我一直想做一个以语音开始、以语音结束的 AI 应用，而不是只停留在文字聊天层面。在中国的ChatAI市场,占有率最大的不是AI性能最好的,而是最先实现用户说,AI回答的豆包.ATA(Audio-to-Audio)的交互方式,肯定是未来交流式AI的主流.

所以 SimsCommu 的背景并不复杂：我想给自己做一个能练口语的系统，同时也拿它验证一条具体的工程路线，也就是 STT 负责听、TTT 负责理解和生成、TTS 负责说。这件事如果跑通，后面很多语音场景都可以在这个基础上继续长出来。

## Implementation

目前的实现已经把这条主链路串起来了。用户先说话，系统把语音送进 STT，再把文本交给对话层LLM处理，最后由 TTS 把回复重新播报出来。为了让练习更接近场景对话，我没有把它做成单角色聊天，而是加入了主持人 Agent ,支持人Agent会分析对话场景设置对应的业务角色，让系统自己判断下一轮应该由谁来回复。

从结构上看，这仍然是一个比较清晰的三段式应用：STT -> TTT -> TTS。不客气的说,在我实现代码的时间点上, 这种构造已经过时,业界已经实现了同时STT和TTS的双工架构.AI可以打断用户的说话,用户也可以随时插话.

## 重启条件

项目重启时将重新设计语音主链路，以 Live 双工模型取代当前的串行架构，再评估主持人 Agent、角色调度和练习反馈应该如何接入实时会话。
